# Planning and Direction System
*Version 1.0 — Initial draft. This document is a living system and will be updated as the platform evolves.*

---

## Overview
The LESARUSS planning system is built to run a platform that continuously evolves. We are not planning quarters in isolation — we are planning version cycles. Every planning period has one core question: **What does the next version of the system need, and what do we need to do to build it?**

Planning is three-layered: annual direction, quarterly version cycles, and weekly execution. Each layer informs the one below it. Direction cascades into milestones, milestones cascade into tasks.

---

## Layer 1 — Annual Direction

At the start of each year, LESARUSS sets its direction across four areas:

### Platform
- What version are we targeting by end of year?
- What major system improvements are planned?
- What new capabilities or tiers might be introduced?

### Growth
- What is the target number of operators in the system?
- What is the target revenue across tiers?
- What is the target number of certified operators?

### Community
- What does a healthy community look like at year end?
- What community infrastructure needs to exist that doesn't exist today?

### Operations
- What team, tools, and processes need to be in place to support the above?

**Output:** Annual Direction Document — one page, four areas, directional targets. Not a rigid plan. A compass.

---

## Layer 2 — Quarterly Version Cycles

Each quarter is organized around a system version milestone. The version cycle has four phases:

### Phase 1 — Collect (Week 1–2)
Review all feedback from the previous quarter: client intake data, operator notes, community activity, version upgrade responses, and any flagged gaps in the system. Identify the 3–5 most meaningful improvements to make.

### Phase 2 — Build (Week 3–10)
Build the improvements. This may include updates to documentation, new templates, refined intake questions, new training materials, or operational changes to how doors are managed. Every build decision is tied back to a real input from the collective.

### Phase 3 — Release (Week 11)
Release the new system version. Notify all users. Deliver upgrade instructions by tier. Create content around what changed and why it matters. Celebrate the community contributions that made the upgrade possible.

### Phase 4 — Observe (Week 12–13)
Watch how operators respond to the new version. Are they applying the upgrades? Are there new questions or gaps emerging? Collect clean data for the next cycle.

**Output:** Each quarter closes with a version number, a release summary, and a feedback file ready for the next cycle.

---

## Layer 3 — Weekly Execution

Weekly planning is tactical. It answers: **What specifically needs to happen this week to stay on track for the quarterly version cycle?**

### Weekly Rhythm

**Monday — Orientation**
- Review where we are in the current version cycle phase
- Confirm top 3 priorities for the week
- Flag any blockers or dependencies

**Wednesday — Midpoint Check**
- Are we on track?
- Is anything blocking a priority?
- Any new inputs from the community or operators that need to be addressed?

**Friday — Close**
- What got done?
- What didn't and why?
- What carries to next week?
- Any new information that should influence the quarterly build?

## Prioritization Framework

When deciding what to work on, LESARUSS uses a simple filter:

1. **Does this directly improve the system for operators?** — Highest priority.
2. **Does this directly support onboarding or delivery?** — High priority.
3. **Does this grow the number of operators in the system?** — High priority.
4. **Does this support operations and team capacity?** — Medium priority.
5. **Does this improve optics but not outcomes?** — Low priority. Defer or cut.

Everything that doesn't fit into one of these categories gets scrutinized hard before it gets time.

## Version Numbering Convention

| Version | Meaning |
|---|---|
| V1.0 | Initial release — foundational system live |
| V1.x | Minor improvements, refinements, and additions within the same core structure |
| V2.0 | Major structural upgrade — new capabilities, new tier logic, or significant framework changes |

Version numbers are communicated publicly to the community. Operators should always know what version of the system they are running.

---

## Alignment Rules

- Every project, task, and initiative should be traceable to a version cycle goal
- If something can't be tied to the current version cycle, it is either a future cycle item or it should not exist
- Feedback from operators is not optional input — it is required input for every version build
- The community is a planning stakeholder. What they are asking for and struggling with shapes what gets built next
